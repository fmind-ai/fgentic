## 5. Bridge State Management (as implemented)

1. **Storage:** the `bridge` CNPG database; one shared `dbutil` pool (pgx driver) backs both the mautrix SQL StateStore and the bridge's own tables. Schema creation is idempotent DDL at startup â€” two tables don't justify a migration framework.
1. **Tables:** `mx_*` (mautrix StateStore, schema owned/upgraded by mautrix); `bridge_contexts(room_id, ghost, context_id, updated_at, PK(room_id, ghost))`; `bridge_processed_events(event_id PK, processed_at)` pruned opportunistically past 24h.
1. **Semantics:** context rows are best-effort (loss degrades to a fresh conversation, never an error). Event dedup is insert-if-absent **before** dispatch, so Synapse's at-least-once transaction delivery collapses to effectively-once agent invocation. A redelivery emits a content-free `duplicate` audit verdict without dispatching again. On a store error the bridge proceeds and records `check_error` (a rare duplicate beats a dropped delegation).
1. **HA note:** with Postgres-backed dedup and per-room ordering keyed in one process, `replicas: 1` is an enforced invariant (the appservice protocol is single-consumer). The Deployment uses a zero-surge rollout (`maxSurge: 0`, `maxUnavailable: 1`), so upgrades never overlap two independent dispatchers; resilience is homeserver-side transaction retry + fast pod rescheduling, not replicas.

## 6. Async Delegation (as implemented)

1. The routing and invocation allowlist is a versioned `agents.yaml` document governed by [agents.schema.json](../apps/matrix-a2a-bridge/agents.schema.json). Current files declare `schemaVersion: 1`; an unknown major fails closed at startup and reload, while an omitted version temporarily defaults to v1 with a structured deprecation warning. Invalid reloads never replace the last-known valid map.
1. `HandleMessage`: drop own bot/ghost senders and anything but `m.text` (D8) â†’ classify the validated full sender MXID against configured external-appservice namespaces â†’ resolve targets under D6's rules â†’ dedup by event ID (D4) â†’ emit a duplicate-delivery audit and stop, or enqueue the immutable `(room, ghost, mapped target, classified sender, prompt)` per target; return immediately. Each target is exactly one local kagent `namespace`/`name` pair or one remote `url`/`timeout`/`tokenBudget`/`cardIdentity` trust pin. A bridged sender is local by MXID but remains denied unless that target's `allowedSenders` explicitly matches it. The dispatcher counts accepted running plus queued jobs and defaults to 32 per room and 256 globally. Capacity exhaustion fails closed before invocation admission: no limiter token, A2A call, or Matrix response is produced; the bridge records `outcome=queue_full`, `terminal_stage=queue`, and the room/global capacity reason.
1. Worker (per-room FIFO, global concurrency cap): atomically re-read the current origin, mapping, and sender policy before spending a limiter token or calling A2A. A removed grant or changed target fails closed; removing an origin rule cannot downgrade an already classified bridged event to native Matrix. For a remote target, the bridge first requires a currently trusted A2A v1.0 AgentCard: its ES256 signature over the JCS-canonical card must verify under the pinned P-256 public JWK, and its protected key ID, name, provider organization, exact JSONRPC or HTTP+JSON endpoint, and token-budget extension must match the configured pins. Startup, config reload, and independent five-minute refreshes revalidate that card. An unsigned, invalid, mismatched, or post-signature-tampered card normally quarantines the target before ghost registration and limiter admission, and emits `outcome=denied`, `terminal_stage=agent_card`, `terminal_reason=agent_card_untrusted`, `rate_limit_verdict=not_checked`, and `a2a_attempted=false`. A per-generation transport guard closes the readiness race: if trust changes after admission, a copied stale client is refused before HTTP with `rate_limit_verdict=allowed` and `a2a_attempted=false`; if trust changes while polling an already-started task, polling stops with `rate_limit_verdict=allowed` and `a2a_attempted=true`. The separate Matrix membership-invite handler may register a configured ghost without delegating to it.
1. All bridge-generated directory, policy-denial, and invocation-rate-limit notices share a response plane with separate per-sender/per-room buckets; exhaustion is silent, so a rejected response cannot amplify a flood. Each invocation/notice sender/room map is independently capped at 4096 keys by default. Unknown keys fail closed at capacity without evicting an active bucket or resetting its burst; idle buckets become reusable through a bounded, at-most-once-per-minute sweep. A denied bridged sender posts at most one bounded policy `m.notice`, emits an origin-attributed audit/metric, and never consumes invocation capacity or calls A2A. An allowed, trusted target then registers/joins its ghost, checks the invocation budgets (D7), shows a **typing indicator** while working, and sends `message/send` with the `(room, ghost)` contextId and Matrix sender in `X-User-Id`. Local calls use the global 60-second transport deadline and bridge workload credential. Remote calls add their configured whole-delegation ceiling to the global `REQUEST_TIMEOUT` and `TASK_TIMEOUT`, never inherit `A2A_API_KEY`, and activate `https://fgentic.fmind.ai/a2a/extensions/token-budget/v1` with `{maxTokens: tokenBudget}` metadata. That scalar is a partner-enforced request contract, not bridge-observed or hard local model-token accounting. Bridged limiter keys include bounded network, full MXID, and agent; metrics keep only bounded ghost/outcome labels.
1. Terminal `Message`/`Task`: post the extracted text (artifacts â†’ status message â†’ last agent turn) as the ghost â€” an **`m.notice` reply** to the original event.
1. Non-terminal `Task`: post a "âŹł working on itâ€¦" placeholder â†’ poll `tasks/get` (2s â†’ 15s exponential backoff, 3-error budget, overall `TASK_TIMEOUT` 10m) â†’ **edit** (`m.replace`) the placeholder into the final answer.
1. Working-state progress: while polling, each new `tasks/get` status message is surfaced as a **threaded** `m.notice` (`m.relates_to` `rel_type: m.thread`, rooted at the placeholder), keeping the main timeline to placeholder â†’ final answer while a collapsible sub-timeline shows detail. Updates are deduplicated (identical consecutive text is dropped) and hard-capped at `MAX_TASK_PROGRESS_POSTS` (default 3; 0 disables) so a flood of status changes never becomes notice-plane pressure (D7). The final answer stays the `m.replace` edit of the root so thread-summary previews remain correct. Optionally (`PIN_IN_FLIGHT_TASKS`, off by default) the placeholder is added to `m.room.pinned_events` while the task runs and removed on any terminal state, giving a zero-UI "what is running here" view; it needs the ghost to hold the room's state-event power level and degrades silently where that power is missing (per-room FIFO dispatch serialises a room's tasks, so the read-modify-write cannot race).
1. Cancel from the room: a member reacts âťŚ to the placeholder of a running long task. The bridge stops polling, calls A2A `tasks/cancel` (best-effort, so token burn halts at the source), edits the placeholder to a content-free "đź›‘ canceled by â€¦" notice, and audits `outcome=canceled`, `terminal_stage=task_cancel`, `terminal_reason=canceled_by_room`, and the `canceled_by` MXID. Authorization: the original delegating sender always may; anyone else needs at least `CANCEL_MODERATOR_POWER_LEVEL` (default 50), read from the cached StateStore only (a cold cache fails closed â€” no network call blocks the synchronous event processor). Other reactions and reactions to unknown events are ignored (untrusted input never invokes an agent).
1. Every ghost-authored event â€” replies, the working placeholder, `m.replace` edits, and the directory/policy/rate-limit notices â€” also carries the MSC3955 `m.automated` mixin so other automation can reliably skip agent output at the content layer rather than by MXID heuristics (federation-safe: no localpart matching). The additive raw key rides alongside the untouched `m.notice`, so mixin-unaware clients render replies unchanged; edits mirror it into `m.new_content` so edit-aware clients keep the marker after applying the replacement. The key is `org.matrix.msc1767.automated: true` â€” MSC3955's "Unstable prefix" section deliberately reuses the parent MSC1767 namespace (which is also what interoperating implementations emit); flip to the stable `m.automated` only once the MSC lands.
1. Failures post a short, generic ghost message (internal endpoints/errors never leak into rooms) and log the full error; empty replies get an explicit "(the agent returned no content)".
1. Shutdown marks the appservice unready and asks the bridge-owned HTTP server for five seconds of graceful quiescence. On timeout it force-closes active transaction connections, so a handler that runs later cannot return a successful acknowledgement and Synapse will retry. Only then does an identity-checked barrier pass behind every event from a transaction capable of acknowledgement before the processor stops. The bridge gives that drain five seconds with delegation contexts live; if exceeded, it cancels those contexts to unblock handlers but still waits for the same barrier rather than discarding acknowledged events. Accepted delegations otherwise keep their runtime context for `SHUTDOWN_TIMEOUT` (25 seconds); then cancellation terminates running work and drops queued work under typed callbacks. Enqueue races and accepted queue drops emit `outcome=shutdown` with `shutdown_enqueue_rejected` or `shutdown_queued_dropped`. The chart's 45-second termination grace covers HTTP intake, event draining, delegation grace, and telemetry cleanup.
