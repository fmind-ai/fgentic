# Bridge Specification (formerly SPEC §5, §6, §12)

## 5. Bridge State Management (as implemented)

1. **Storage:** the `bridge` CNPG database; one shared `dbutil` pool (pgx driver) backs both the mautrix SQL StateStore and the bridge's own tables. Schema creation is idempotent DDL at startup — two tables don't justify a migration framework.
1. **Tables:** `mx_*` (mautrix StateStore, schema owned/upgraded by mautrix); `bridge_contexts(room_id, ghost, context_id, updated_at, PK(room_id, ghost))`; `bridge_processed_events(event_id PK, processed_at)` pruned opportunistically past 24h.
1. **Semantics:** context rows are best-effort (loss degrades to a fresh conversation, never an error). Event dedup is insert-if-absent **before** dispatch, so Synapse's at-least-once transaction delivery collapses to effectively-once agent invocation. A redelivery emits a content-free `duplicate` audit verdict without dispatching again. On a store error the bridge proceeds and records `check_error` (a rare duplicate beats a dropped delegation).
1. **HA note:** with Postgres-backed dedup and per-room ordering keyed in one process, `replicas: 1` is an enforced invariant (the appservice protocol is single-consumer). The Deployment uses `Recreate`, so upgrades never overlap two independent dispatchers; resilience is homeserver-side transaction retry + fast pod rescheduling, not replicas.

## 6. Async Delegation (as implemented)

1. `HandleMessage`: drop own bot/ghost senders and anything but `m.text` (D8) → classify the validated full sender MXID against configured external-appservice namespaces → resolve targets under D6's rules → dedup by event ID (D4) → emit a duplicate-delivery audit and stop, or enqueue the immutable `(room, ghost, mapped path, classified sender, prompt)` per target; return immediately. A bridged sender is local by MXID but remains denied unless that target's `allowedSenders` explicitly matches it. The dispatcher counts accepted running plus queued jobs and defaults to 32 per room and 256 globally. Capacity exhaustion fails closed before invocation admission: no limiter token, A2A call, or Matrix response is produced; the bridge records `outcome=queue_full`, `terminal_stage=queue`, and the room/global capacity reason.
1. Worker (per-room FIFO, global concurrency cap): atomically re-read the current origin, mapping, and sender policy before spending a limiter token or calling A2A. A removed grant or changed path fails closed; removing an origin rule cannot downgrade an already classified bridged event to native Matrix. All bridge-generated directory, policy-denial, and invocation-rate-limit notices share a response plane with separate per-sender/per-room buckets; exhaustion is silent, so a rejected response cannot amplify a flood. Each invocation/notice sender/room map is independently capped at 4096 keys by default. Unknown keys fail closed at capacity without evicting an active bucket or resetting its burst; idle buckets become reusable through a bounded, at-most-once-per-minute sweep. A denied bridged sender posts at most one bounded policy `m.notice`, emits an origin-attributed audit/metric, and never consumes invocation capacity or calls A2A. An allowed target then registers/joins its ghost, checks the invocation budgets (D7), shows a **typing indicator** while working, and sends `message/send` with the `(room, ghost)` contextId, Matrix sender in `X-User-Id`, and a 60-second transport deadline. Bridged limiter keys include bounded network, full MXID, and agent; metrics keep only bounded ghost/outcome labels.
1. Terminal `Message`/`Task`: post the extracted text (artifacts → status message → last agent turn) as the ghost — an **`m.notice` reply** to the original event.
1. Non-terminal `Task`: post a "⏳ working on it…" placeholder → poll `tasks/get` (2s → 15s exponential backoff, 3-error budget, overall `TASK_TIMEOUT` 10m) → **edit** (`m.replace`) the placeholder into the final answer.
1. Failures post a short, generic ghost message (internal endpoints/errors never leak into rooms) and log the full error; empty replies get an explicit "(the agent returned no content)".
1. Shutdown marks the appservice unready and asks the bridge-owned HTTP server for five seconds of graceful quiescence. On timeout it force-closes active transaction connections, so a handler that runs later cannot return a successful acknowledgement and Synapse will retry. Only then does an identity-checked barrier pass behind every event from a transaction capable of acknowledgement before the processor stops. The bridge gives that drain five seconds with delegation contexts live; if exceeded, it cancels those contexts to unblock handlers but still waits for the same barrier rather than discarding acknowledged events. Accepted delegations otherwise keep their runtime context for `SHUTDOWN_TIMEOUT` (25 seconds); then cancellation terminates running work and drops queued work under typed callbacks. Enqueue races and accepted queue drops emit `outcome=shutdown` with `shutdown_enqueue_rejected` or `shutdown_queued_dropped`. The chart's 45-second termination grace covers HTTP intake, event draining, delegation grace, and telemetry cleanup.

## 12. Testing & Validation Ladder

1. **Unit — done:** the race-enabled suite covers target resolution (federation spoofing, sender policy, dedup), dispatch ordering/concurrency/shutdown, rate-limit config, state semantics, and the `Task|Message` result mapping, including non-terminal tasks. The repository's `mise run check` gate verifies lint, vulnerabilities, formatting, secrets, rendered manifests, Terraform, workflows, and IaC configuration.
1. **Contract — done:** the in-process `a2asrv` fixture exercises empty and terminal `Message` replies, terminal and working `Task` results, contextId reuse, request deadlines, transient `tasks/get` recovery with capped exponential backoff, and the literal kagent-compatible A2A wire version. Because the fixture uses the SDK's server half but asserts the expected version independently, an SDK default change cannot silently move both client and fixture together.
1. **Integration — done:** the isolated kind fixture starts Postgres, Synapse, the real bridge, and an SDK-backed A2A server; registers the appservice; creates and invites a room; drives `@mention → m.notice` through the Matrix client API; and replays the transaction to prove exactly-once delegation. CI runs it for bridge and fixture changes.
1. **E2E demo script (Phase 5):** the runbook-driven "enterprise showcase" path: bootstrap → login → room → mention → reply → one Grafana trace covering the full hop. The demo _is_ the acceptance test.
1. **Load sanity (§12.5) — done:** the isolated kind regression drives 100 mentions across ten rooms through real Synapse appservice transactions and a 2 s SDK-backed A2A stub. It asserts prompt transaction acknowledgement, the 16-delegation global cap, bounded heap, queue drain, per-room FIFO starts and completions, and exactly one reply per mention after transaction replay. See the [dated reference measurements and re-run command](performance.md).
