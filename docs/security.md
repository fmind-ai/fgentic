## 7.1 Self-hosted model egress proof

The vLLM profile separates supply-chain bootstrap from the prompt data plane. A revision-pinned loader Job may use cluster DNS and public TCP 443 to fetch public model weights into a PVC; it has no Matrix, agent, or prompt input and completes before serving becomes Ready. The serving Pod mounts that cache read-only, disables usage telemetry and online model resolution, and has no NetworkPolicy egress rule. A profile-scoped policy also prevents the agentgateway proxy from reaching anything except cluster DNS, its XDS control plane, vLLM, and the existing kagent A2A backend.

Manifests are intent, not enforcement evidence. Repo-owned k3d servers deliberately disable K3s's embedded kube-router NetworkPolicy controller: this constrained shared host aborts `iptables-restore` with `sendmsg() failed: Message too large`, leaving no pod firewall chains while repeatedly logging failed rulesets. `mise run test:network-policies:kind` separates that host limitation from manifest correctness: it creates an isolated kind cluster with checksum-pinned Calico, applies the exact policies, proves each required deny/allow path (including vLLM serving egress denial), and verifies that deleting `kagent-allow-platform` turns the test red. This is deterministic policy evidence, not acceptance of the local k3d cluster or any target deployment. GKE production acceptance requires Dataplane V2 evidence. Before claiming that prompts never leave an installed cluster, repeat the active-cluster probe and capture the complete Matrix mention-to-reply flow without external LLM connections.

## 7.2 Attribution is not downstream authentication

The [delegation attribution runbook](audit.md) defines the evidence chain from Matrix sender to the bridge's `X-User-Id` assertion, kagent session/task metadata, agentgateway request logs, and aggregate Prometheus series. A local Synapse event identifies the authenticated Matrix session that submitted it; after the bridge, kagent 0.9.11's unsecure mode accepts the caller-supplied header and marks the resulting user unauthenticated. The bridge/kagent NetworkPolicies are therefore load-bearing controls against a bypassing caller, subject to installed-cluster CNI acceptance in §7.1.

Agentgateway and Prometheus do not currently receive the Matrix event, sender, A2A context/task, or kagent invocation identity on model requests. They corroborate provider/model/token activity but cannot uniquely identify a user or task under concurrency. Treat the bridge's content-free structured record plus kagent's matching session/task as attribution evidence; never treat the model-facing provenance prompt, gateway time correlation, or a high-cardinality metric label as identity proof.

[D18](design-decisions.md) does not promote `X-User-Id` into authentication or change this global boundary. It defines a separate, typed identity-and-audience projection for the exact permission-aware retrieval path. That projection remains future behavior until [#333](https://github.com/fmind-ai/fgentic/issues/333) implements and proves every workload, admission, ext-auth, database, and NetworkPolicy control in [ADR 0017](adr/0017-permission-aware-identity-binding.md).

## 7.3 A2A workload authorization

The pinned agentgateway v1.3.1 source defines strict API-key authentication, exposes Secret metadata under `apiKey`, and translates `Require` CEL expressions into fail-closed authorization rules. It selects one `AgentgatewayPolicy` for an HTTPRoute target, so the selected provider profile owns one combined workload-and-model-admission policy rather than two conflicting resources. Fgentic gives only the bridge a random A2A credential; its server-side copy carries `workload: matrix-a2a-bridge`. The policy additionally limits that identity to `POST` under `/api/a2a/kagent/` and AgentCard `GET` requests ending in `/.well-known/agent-card.json`. A missing or invalid credential returns 401; a valid credential with the wrong workload, method, or path returns 403.

For `POST`, the same policy requires the bridge-reviewed `X-Fgentic-Data-Classification` value and admits it only when the exact selected model's governed ceiling covers that class. `agents.yaml` accepts only `public`, `approved_non_public`, `restricted`, or `regulated`; omission defaults to `regulated`, and the local transport overwrites the header. The remote transport never receives it. The catalog validator derives and checks each profile's committed policy expression, so an unknown exact model or uncataloged profile denies `SendMessage`. This check sits on the authenticated bridge A2A boundary: kagent 0.9.11 starts a separate model request and does not safely propagate arbitrary A2A headers, so no Agent-supplied model header is treated as trusted policy. The vLLM-selected profile admits all governed classes and limits proxy model egress to its internal serving Pod; the Vertex and demo entries admit only `public`.

This is defense in depth over NetworkPolicy, not a substitute for it. The credential distinguishes the bridge workload from other pods even if ingress isolation is accidentally widened, while NetworkPolicy limits which workloads can attempt a connection and protects kagent's otherwise unauthenticated direct endpoint. One workload-wide bridge credential cannot express a different gateway identity per mapped agent; the bridge mapping and Matrix room membership remain the per-agent authorization boundaries. D18 makes agentgateway ext-auth the local identity-projection root only for a separately admitted retrieval-capable Agent, and disables that Agent's direct-kagent fallback; ordinary A2A behavior remains unchanged.

`mise run check:a2a-authorization` verifies the workload manifest, Secret metadata, and CEL contract. `mise run test:model-residency` renders every provider, checks the catalog-derived policy, proves the public-only API ceiling and regulated vLLM ceiling, and verifies vLLM has no external model-egress destination. `mise run test:a2a-authorization` runs unauthorized, wrong-identity, wrong-path, and valid requests against the exact agentgateway image version pinned by Flux. Target-cluster acceptance still combines those contracts with `mise run test:network-policies` so the direct kagent bypass is tested independently.

## 7.4 Governed MCP tool egress

The pinned kagent 0.9.11 source at commit `14dcbfc49bde990a0c737da14a9d94cf7e788303` resolves a `RemoteMCPServer` into the managed Agent pod's runtime configuration. `go/core/internal/controller/translator/agent/adk_api_translator.go` shows that, without `proxy.url`, that pod dials the resource's URL directly; with the configured URL it preserves `/mcp`, changes the destination to the proxy, and adds `x-kagent-host` for the original internal Service. `go/core/internal/controller/reconciler/reconciler.go` separately shows that the controller still dials `kagent-tools:8084` directly to discover tools. The `agent-zoo-egress` policy selects only managed zoo pods, blocks their direct `:8084` path, and retains `:8083` for their required session/persistence API.

agentgateway v1.3.1 at commit `dbaaf7ed73671e7aec9195e35e7f726c0b14b84a` terminates that MCP connection, authenticates platform-helper with a random SOPS-managed bearer key, and binds server-side `apiKey.agent=platform-helper` metadata to it. A fail-closed MCP authorization policy exposes only `k8s_get_resources`, `k8s_describe_resource`, `k8s_get_events`, `k8s_get_resource_yaml`, and `k8s_get_pod_logs`; the upstream server independently runs read-only and has get/list/watch RBAC in six named namespaces with no Secret or cluster-scoped access. docs-qa and scribe have neither a tool declaration nor this credential.

The current MCP path therefore carries only the authenticated Agent identity and static credential; it carries no verified Matrix principal, partner-client mapping, or room audience. kagent v0.9.11 can propagate an explicit request-header allowlist, but Fgentic's admission policy currently rejects all such propagation. Issue #333 must narrow that rule to exactly `X-Fgentic-Identity`, the retrieval Agent, and the knowledge-retrieval server, then add the A2A ext-auth projection, MCP workload/path/tool authorization, and database prefilter before any grounded-content claim is made.

The upstream `kagent-tools` image is immutable by digest. `infra/agentgateway/mcp-surface.pin.json` separately records its container command, ordered runtime arguments, discovery and governed-backend routes, and complete current typed MCP surface: the initialize response plus tools, prompts, resources, and resource templates, with unsupported and supported-empty collections kept distinct. Initialization is pinned because its model-facing `instructions`, server identity, capabilities, metadata, and negotiated protocol can be poisoned before any list call. Every full object is JCS-canonicalized and hashed, so drift in instructions, descriptions, JSON Schemas, annotations, titles, icons, or `_meta` requires an explicit pin update in a reviewed PR. Sorting identities and following every cursor make response ordering irrelevant. The only server-specific normalization sorts the nine identifiers in `k8s_generate_resource.inputSchema.properties.resource_type.description`, which upstream constructs by iterating a Go map; a changed identifier, prose, path, or surrounding schema still produces drift.

`infra/mcp-catalog/` is the complementary first-review boundary. Its versioned official-registry `server.json` entry records the stable source identity and license, immutable image and source revision, approving maintainer group/date, complete surface-pin hash, five allowed tools, and the exact agentgateway/kagent resources that consume it. Both resources carry `fgentic.dev/mcp-catalog-entry=kagent-tools`; the chart-owned `RemoteMCPServer` receives that identity through the Flux Helm post-renderer. The offline catalog check rejects missing or unknown annotations, incomplete resource coverage, a tag-only image, a changed pin, or an allowed tool absent from the reviewed surface. The catalog remains plain git data rather than another writable runtime service; its [vetting checklist](../infra/mcp-catalog/README.md) is the operator review path.

The pin deliberately retains `destructiveHint: true` and `openWorldHint: true` on all eight current upstream tools; the typed encoding omits a false/default `readOnlyHint`. Read-only enforcement comes from the immutable `--read-only` process argument, namespaced RBAC, and the five-tool gateway allowlist—not from trusting those annotations.

This key is a narrow, rotatable application credential—not a non-exportable pod identity, mTLS certificate, SPIFFE SVID, Matrix-human identity, or approval token. A compromised platform-helper pod can reuse its own key and any of its five allowed operations. NetworkPolicy does not authenticate a process, and enforcement depends on the target CNI. The controller's direct discovery connection is administrative metadata traffic, not an agent tool call, and remains a separate trusted path. kagent 0.9.11 multiplexes its own `/mcp` handler with session APIs on controller port `8083`; no sample Agent references that handler as a tool, but standard Kubernetes NetworkPolicy cannot allow the required session paths while denying a different HTTP path. Removing that residual reachability needs an upstream separate listener or authenticated session proxy, not an inaccurate L4 policy claim.

For each parsed `tools/call`, agentgateway emits a JSON stdout record marked `audit.kind=fgentic.mcp_tool_call.v1` with authenticated agent, MCP method, resolved tool name/target, status, and duration. The policy adds no arguments, results, prompts, Matrix content, or authorization headers. Kubernetes stdout is not durable retention; production evidence requires an approved restricted sink. Inspect the content-free fields without printing other logs:

```bash
kubectl -n agentgateway-system logs deploy/agentgateway-proxy --since=15m \
  | jq -R '
      fromjson?
      | select(.["audit.kind"] == "fgentic.mcp_tool_call.v1")
      | {
          time,
          agent: .["fgentic.agent"],
          method: .["mcp.method"],
          tool: .["mcp.tool.name"],
          target: .["mcp.tool.target"],
          status: .["http.status"],
          duration
        }'
```

`mise run check:mcp-governance` checks the rendered proxy, credential copies, route, policies, exact tool set, direct-egress omission, audit schema, pin integrity, backend/Service coverage, and complete execution/routing provenance without Docker or a live cluster/server; cold chart or module caches can still require network access. `mise run test:mcp-governance` first compares the live read-only server with that reviewed pin, then uses the exact pinned agentgateway image plus a deliberately write-capable disposable kagent-tools fixture to prove wrong agents are rejected, write tools are filtered, an allowed call reaches the backend, and arguments/credentials stay out of the audit record. `mise run test:network-policies:kind` independently proves the direct managed-agent `:8084` denial and gateway `:8084` allow on Calico. Target-cluster acceptance repeats those paths on the installed CNI, then captures one real platform-helper call and its audit record.

The pin is a review and reconcile boundary, not a content verdict. It cannot detect malicious-from-day-one metadata, prove that implementation behavior matches metadata, or replace authorization and catalog vetting. The official SDK preserves all current typed fields but intentionally drops unknown future typed fields; open maps such as `_meta`, schemas, and experimental or extension capabilities remain complete. Upgrade the versioned pin format when the protocol adds relevant fields. agentgateway v1.3.1 filters and denies MCP identifiers at runtime but has no native committed-hash comparison, and adding an always-on external guardrail only for that comparison would widen the trusted runtime. Scheduled live verification therefore detects post-review drift; the catalog records the independent human approval boundary for first-review evidence.

## 7.5 Cross-organization A2A authorization

The federation profile exposes one deliberate exception to the cluster-internal A2A rule: `POST https://a2a.org-a.fgentic.localhost/api/a2a/kagent/docs-qa`, plus its anonymous AgentCard `GET`. The external Gateway accepts only those exact method/path pairs from `agentgateway-system`; agentgateway handles them on a separate `federation-a2a` listener. The backend remains the internal kagent controller Service, and no route publishes another Agent or exposes kagent directly.

The card and the caller answer different security questions. An ES256 JWS over the RFC 8785 JCS-canonical card, verified with the pinned public P-256 JWK and protected `kid`, establishes the declared docs-qa identity and detects tampering. It does not authorize an invocation. The lab stores that verify-only JWK as a non-secret evidence artifact; production onboarding must deliver and pin it through an independently authenticated channel rather than trusting a key learned from the card endpoint itself. The lab's implemented transport policy instead validates a five-minute org-B `client_credentials` JWT against the exact issuer, remote JWKS, `fgentic-a2a` audience, and `azp=org-b-a2a`. The resulting identity is the OAuth client, not a Matrix MXID, employee, or workload instance. The card stays public so consumers can evaluate identity and capabilities before authenticating.

For every `SendMessage`, authorization also requires the exact `A2A-Version: 1.0` service parameter, the token-budget extension, and an integer `maxTokens` within the configured ceiling. The version guard prevents a v1 card from silently invoking kagent's compatibility-default v0.3 JSON-RPC binding. The global rate-limit service uses the verified `azp` as its descriptor and `maxTokens` as admission cost, failing closed when quota cannot be checked. This reserves a caller-declared maximum; it is not provider-reported consumption. Actual model-token metrics arise on the later kagent → agentgateway model hop and remain aggregate, so they cannot support per-consumer billing or prove which A2A task used a token.

mTLS is the supported alternative when partners operate certificate identity rather than OIDC: terminate on a dedicated client-certificate listener, trust only the partner CA, map an exact SAN to the quota consumer, and define issuance, rotation, and revocation. The lab does not implement or test that path, and ordinary server-side TLS is not mTLS. Whichever pattern a deployment selects, keep the exact route, fail-closed policy, per-consumer quota, and direct-kagent NetworkPolicy invariant.

## 7.6 Admission-enforced platform invariants

The cluster installs `infra/policies/` before creating any managed namespace. Kubernetes' in-process ValidatingAdmissionPolicy then rejects four classes of drift without adding a webhook controller:

1. Pods in a namespace labelled `fgentic.dev/managed=true` cannot use an explicit `:latest` tag or an untagged reference that implicitly resolves to latest in a normal, init, or ephemeral container.
1. A managed namespace must declare `enforce`, `audit`, and `warn` Pod Security levels plus its image-policy phase. PSS version labels are forbidden so the levels always use the current standard. The PSS values and managed marker cannot later be changed or removed through the API; image policy may move from `audit` to `enforce`, but never backwards.
1. A `kagent.dev/v1alpha2` Agent must live in `kagent`, be Declarative, use only `agentgateway-model`, keep the Python runtime, and run under the reviewed `agent-zoo-runtime` pod identity without image, container, volume, scheduling, security-context, or service-account overrides. Its deployment environment is a narrow exact contract: it must contain one literal `false` entry for each of `ADK_CAPTURE_MESSAGE_CONTENT_IN_SPANS`, `OTEL_INSTRUMENTATION_GENAI_CAPTURE_MESSAGE_CONTENT`, and `TRACELOOP_TRACE_CONTENT`, with no `valueFrom` or additional entry. Missing, changed, or duplicated controls are rejected. Secondary memory/compaction models and Secret-backed prompts are rejected. Only `platform-helper` may declare a tool, and that one reference is limited to the reviewed same-namespace RemoteMCPServer, five read-only tool names, and its exact Authorization Secret. Inline credentials and request-header propagation are rejected. BYO, Agent-as-tool, sandbox, legacy top-level skills, and model-generated code execution are outside the admitted surface.
1. Managed namespaces admit only ClusterIP or ExternalName Services. The exact `gateway/traefik` LoadBalancer is the sole public-service exception; NodePort and `externalIPs` are rejected everywhere else.

The digest rule is deliberately staged rather than falsely platform-wide. The `models`, `trivy-system`, and optional `bridges` namespaces contain only digest-pinned owned workloads and use `fgentic.dev/image-policy=enforce`, so tag-only Pods are denied. Other namespaces use `audit`: upstream Helm charts still emit tag-only image references, and the supported local bridge side-load is intentionally `matrix-a2a-bridge:local`. Those requests receive a client warning and an audit annotation; they must move to `enforce` only after every controller-generated Pod survives a fresh rollout. `Warn` keeps the debt client-visible, but the selected low-noise API audit policy omits ordinary Pod writes, so it does not collect those admission annotations and makes no durable evidence claim for them.

Every CEL policy uses `failurePolicy: Fail`, and its annotation records the simple constant or linear cost shape. Namespace selectors exclude `default`, `flux-system`, `kube-system`, `kube-public`, and `kube-node-lease` because Fgentic does not label or claim ownership of them. This is scoping, not a bypass for platform workloads. API-created admission policies cannot intercept mutation of ValidatingAdmissionPolicy or its bindings themselves; Flux drift detection and repository review remain that control. CEL also cannot prove cross-object invariants such as a namespace having a NetworkPolicy, so the independent conformance gate remains load-bearing.

`mise run check:admission-policies` checks the policy set, staged bindings, managed namespace profiles, and policy-before-namespace Flux edge offline. With an explicitly approved Kubernetes 1.30+ test context, `ADMISSION_POLICY_CONTEXT=<context> mise run test:admission-policies` performs server-side negative probes without starting or pulling a workload: missing or weakened PSS labels, explicit or implicit latest images across every container class, tag-only images in enforce mode, NodePort, `externalIPs`, wrong Agent models, missing, changed, duplicated, `valueFrom`-backed, or additional trace-content controls, other pod overrides, Agent-as-tool, MCP target/tool expansion, request-header propagation, and credential substitution are rejected; tag-only audit mode warns; pinned Pods, ClusterIP, and all three repository Agents pass. `demo:up` runs that same contract after a clean reconcile and before the mention-to-reply seed.

MutatingAdmissionPolicy is intentionally absent while the k3s pin remains 1.35; reconsider its stable API after moving to Kubernetes 1.36. Admission mutation would complement these validations, not replace explicit GitOps manifests.

## 7.7 Database change audit boundary

The shared PostgreSQL cluster audits only pgAudit's `DDL` and `ROLE` classes. `pgaudit.log_catalog`, `pgaudit.log_parameter`, and `pgaudit.log_statement` are all off: catalog probes stay quiet, and schema/privilege evidence cannot copy SQL literals, bound parameters, prompts, Matrix content, credentials, or row data into the log. Normal `READ` and `WRITE` traffic is excluded both for privacy and to prevent Synapse volume from turning the control into a storage or availability risk.

This control detects database schema and privilege activity under a PostgreSQL session role; it does not identify one pod, workload, Matrix user, or human, and ROLE records may not identify the target role in their object field. pgAudit uses PostgreSQL's best-effort standard logging path: a record can exist for a rolled-back statement or be lost before durable export, and the trail is not trustworthy against a hostile superuser who can change settings or logging. Tenant roles have no superuser access, while superuser/operator access remains a separate restricted administrative boundary.

CloudNativePG converts the audit stream to JSON stdout, but the repository currently provides no durable log sink. The minimal SQL/payload-suppressed projection and validation gates are documented in the [attribution runbook](audit.md); #157 owns restricted transport, retention, and query access. `pg_stat_statements` remains out of scope because query-text and performance-statistics governance require an independent decision rather than being bundled into an audit control.

## 7.8 Kubernetes API audit boundary

The local k3d reference injects [`infra/k3d-audit-policy.yaml`](../infra/k3d-audit-policy.yaml) before k3s starts and enables the API server's JSON log backend at `/var/log/kubernetes/audit/audit.log`. The policy uses `Metadata` for selected object writes and one-object reads, and `None` for everything else. It never uses `Request` or `RequestResponse`: those levels can persist Secret, ConfigMap, admission-policy, or Agent bodies. `RequestReceived` is omitted to avoid duplicate events; list/watch traffic is excluded to keep the single-node profile bounded. Rotation starts at 10 MiB, retains at most three backups, and expires old backups after seven days.

`Metadata` suppresses request and response bodies; it does not mean the entire event is content-free. The audit event retains its request URI, user agent, source addresses, authorization annotations, and object reference. In particular, Kubernetes serializes `pods/exec` command arguments into `requestURI` query parameters. Never put credentials or other sensitive material in exec arguments, restrict audit-log readers, and require any #157 collector to preserve this stream's sensitive-metadata classification.

The retained principal is an API identity, not automatically a person. Direct local administration normally identifies the kubeconfig client. GitOps changes identify Flux's kustomize-controller service account, so the human author and reviewed intent remain a Git-history question. The API trail independently proves when that declared object reached the API server, the verb/target/source, and whether the request succeeded. A cluster administrator who controls the API server, policy, node, or Docker daemon can alter or delete this best-effort local trail.

GKE has a different managed boundary. [Admin Activity audit logs are always enabled](https://cloud.google.com/kubernetes-engine/docs/how-to/audit-logging), while Data Access logs require an explicit Google Cloud logging configuration and may incur cost. GKE supplies the Kubernetes API server's [audit policy](https://cloud.google.com/kubernetes-engine/docs/concepts/audit-policy); this repository cannot replace it with the local file. GKE treats Secrets and ConfigMaps as metadata-only special cases, but operators must configure IAM, log routing, retention, and any required Data Access categories separately before claiming parity with a local read query. The local node path is therefore an input only for the future #157 self-hosted pipeline, never a GKE collection path.

`mise run check:kubernetes-audit` validates the body-suppressed policy and exact canonical k3s flags. `mise run test:kubernetes-audit` proves real file rotation plus a Secret patch/read record and the absence of its value in a disposable no-port k3d cluster; it does not inspect or mutate the shared cluster.

## 7.9 Continuous image vulnerability boundary

Trivy Operator extends D13 from build time to images that remain deployed while vulnerability feeds evolve. It performs registry image analysis; it does not inspect a mutated running filesystem, prove exploitability, detect malware, or replace signature/admission verification. Only vulnerability reports and aggregate finding metrics are enabled. SBOM generation, configuration audit, exposed-secret scan, RBAC and infrastructure assessment, compliance, image-information metrics, and per-CVE metrics are off, which bounds both permissions and metric cardinality.

The operator is confined to the 13 target namespaces listed in [§9.9](observability.md#99-runtime-image-vulnerability-drift). A reusable read/report role is bound separately in each target; it excludes Secret and ServiceAccount access. The chart's broad generated role is disabled, its three aggregate report-view roles are deleted after rendering, and the only cluster-wide grant is read-only list/watch access to the two cluster report kinds that the metrics controller watches. Job and Secret creation, fixed Secret and pod-log reads, event writes, leader election, and the exact cache reads needed for centralized Jobs stay in restricted `trivy-system`, where no observability credentials exist. There is no target report-mutation binding in that namespace, and scan Jobs do not mount a ServiceAccount token.

The operator and scan containers run as UID/GID 10000 with `RuntimeDefault` seccomp, no privilege escalation, every capability dropped, and read-only root filesystems. The operator requests 50m CPU, 64 MiB memory, and 128 MiB ephemeral storage, with 200m, 256 MiB, and 512 MiB limits. Each scanner requests 100m, 128 MiB, and 256 MiB, with 500m, 512 MiB, and 2 GiB limits. v0.32.0's cache-based concurrency check is not an atomic admission boundary; `trivy-system` therefore has an independent ResourceQuota with `count/jobs.batch: 1`, which serializes creation atomically. The disposable runtime test pre-occupies that quota with a suspended hold Job, proves an operator scan Job is rejected, releases the hold, and then requires eventual reports for both fixtures with no concurrent execution. Limits remain safety ceilings rather than capacity evidence. Acceptance also requires the test to record the operator's observed working set during a real scan and show it below the configured limit; no observed baseline is asserted until that gate passes.

Every Pod in `trivy-system` is default-denied. Only Pods carrying the reviewed scanner label accept metrics from `monitoring` on TCP 8080 and may reach cluster DNS plus TCP 443 for the Kubernetes API, image registries, and vulnerability feeds. This port-based policy does not authenticate destinations. A deployment requiring domain-level restriction must use a governed egress proxy or FQDN-aware CNI and prove the effective deny/allow paths.

Private registries are unsupported in this pinned profile, not merely disabled. In v0.32.0 the operator creates an aggregate credential Secret before its scan Job, but a serialization-quota rejection leaves that Secret without an owner; the next reconcile treats `AlreadyExists` as success and may never retry the Job. Enabling `privateRegistryScanSecretsNames` would therefore create a credential-retention and coverage gap precisely on the concurrency path this design hardens. Do not add registry credentials, bind workload Secret access, enable `accessGlobalSecretsAndServiceAccount`, or relax the quota. Support requires an upstream create/order/retry fix plus a runtime case proving cleanup and eventual reporting. A public registry or proxy on a port other than 443 still needs a reviewed NetworkPolicy extension; because standard NetworkPolicy cannot constrain a hostname, pair that exception with a governed egress proxy or FQDN-aware CNI. The [production runbook](production.md#operate-continuous-vulnerability-scanning) owns runtime proof and remediation.

## 7.10 Permission-aware retrieval identity and audience boundary

[ADR 0017](adr/0017-permission-aware-identity-binding.md) and D18 settle the implementation contract; they do not claim it is deployed. The knowledge-retrieval service owns one parameterized database prefilter. agentgateway derives a canonical `X-Fgentic-Identity` from either the authenticated Matrix bridge plus hot-read room state or the exact validated partner `(issuer, audience, azp)` mapping; kagent may propagate only that one header to the exact retrieval MCP route. The projector separately overwrites `X-User-Id` for kagent attribution/session partitioning, never retrieval authority. The MCP gateway binds the one carrier to the exact Agent key, path, and tool; the default-deny retrieval service strictly decodes it before querying.

The boundary uses latest stable A2A specification 1.0.1 (AgentCard interface `protocolBinding: "JSONRPC"` with `protocolVersion: "1.0"`, plus `A2A-Version: 1.0`; JSON-RPC itself remains 2.0) and stable MCP revision `2025-11-25`, not a draft or release candidate. The retrieval server uses Streamable HTTP, negotiates exactly `2025-11-25` during initialization, requires `MCP-Protocol-Version: 2025-11-25` thereafter, and exposes JSON Schema 2020-12 tool schemas. The existing `kagent-tools` MCP `2025-06-18` surface is not the retrieval contract.

For Matrix output, authorization is the intersection of every effective reader's typed full MXID, not merely the initiating sender's access; Matrix group mapping is absent in v1. The effective audience excludes only the bridge-owned bot/ghost accounts inside the trusted delivery workload; external-appservice and separately operated service identities remain readers. The bridge snapshots joined, invite-only, joined-history room state plus its exact room-classification policy before delegation and rechecks the digest immediately before every content-bearing send, retry, edit, or artifact post. Drift or incomplete state suppresses and scrubs the result. Direct partner A2A uses only namespaced groups and classifications from the configured exact JWT-client mapping and supplies no local-principal operand.

This first contract deliberately relies on authenticated workload credentials, narrow admission, and enforced NetworkPolicies; the plain projection does not resist a compromised retrieval Agent pod holding its own MCP key. It therefore rejects missing/unknown classifications and every restricted, regulated, secret, or authentication chunk at the database predicate. Each initial call omits caller context, task, and task-reference IDs so old grounded content cannot bypass a new decision. Matrix delivery uses a conservative 540-second outer deadline. Public OAuth retrieval is synchronous, admits only tokens with more than its shared 1–60-second request timeout plus 30 seconds remaining, and offers no task read/cancel route because pinned kagent does not bind task ownership to the client. The installed-CNI negative probes and the re-evaluation triggers in ADR 0017 are part of correctness, not optional hardening.
