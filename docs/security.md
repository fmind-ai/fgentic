# Security Boundaries & Threat Model (formerly SPEC §7)

Trust zones, outermost first:

| Zone                                              | Trust level                   | Controls                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                               |
| ------------------------------------------------- | ----------------------------- | -------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| Internet → Gateway (Element/Synapse/MAS/Keycloak) | Untrusted                     | Gateway API + TLS; MAS OIDC; Keycloak brute-force protection + default-deny NetworkPolicy; Synapse rate limits. The federation profile replaces broad route ownership with namespace-bound Matrix listeners and one exact `a2a.<org-A>` listener owned by agentgateway.                                                                                                                                                                                                                                                                                                                |
| Federated homeservers (M8)                        | Semi-trusted (contract-bound) | Foreign look-alike rejection and deny-by-default bridge sender policies (D6); closed Synapse allowlists; room-v12 creation; initial server ACLs; and a fail-closed, git-reloadable callback border with content-free denial evidence (§8). These controls restrict admission but cannot make an admitted partner trustworthy or retract replicated history.                                                                                                                                                                                                                            |
| Partner A2A consumer → public delegation listener | Untrusted machine client      | The opt-in federation profile publishes only docs-qa through exact routes. An anonymous ES256/JCS Signed AgentCard establishes the advertised agent identity; strict JWT issuer/audience/`azp` checks authorize org B; a required `maxTokens` reservation is rate-limited per verified `azp`; direct kagent remains ClusterIP-only behind NetworkPolicy. mTLS is a documented alternative, not a second lab claim (§7.5, §8.3).                                                                                                                                                        |
| External network → optional mautrix appservice    | Untrusted identity provider   | Slack/Telegram units are opt-in and digest-pinned with exclusive namespaces, scoped databases, restricted workloads, SOPS credentials, pod-selected internal NetworkPolicy paths, exact agent relay permissions, and an admin-only per-portal relay. A recognized bridged MXID remains denied until an agent's `allowedSenders` explicitly matches it; audit records carry bounded origin kind/network fields. Standard NetworkPolicy still admits arbitrary non-private IPv4 TCP/443 rather than provider FQDNs. Live provider, CNI, and bidirectional acceptance are external gates. |
| Room content → agents                             | **Untrusted input**           | Prompt injection remains unsolved; the [controls and honest limits](security/prompt-injection.md) are provenance, deterministic authorization, least-privilege tools, external approval for consequential actions, and treating every agent/tool output as data                                                                                                                                                                                                                                                                                                                        |
| Bridge → kagent A2A                               | Cluster-internal              | Strict agentgateway API-key authentication + fail-closed workload CEL policy; the exact pinned gateway passes 401/403/200 runtime probes. NetworkPolicies on agentgateway and kagent pass isolated Calico conformance (D11), while installed-cluster acceptance remains required because kagent itself is unauthenticated.                                                                                                                                                                                                                                                             |
| Agents → tools                                    | Governed capability           | kagent rewrites platform-helper's MCP endpoint to agentgateway; a scoped per-agent API key and fail-closed MCP CEL policy admit exactly five read-only Kubernetes operations. The exact pinned gateway passes auth/filter/audit runtime probes, and isolated Calico denies direct agent `:8084` while allowing the gateway; installed-CNI and live-call acceptance remain.                                                                                                                                                                                                             |
| Agents → LLM                                      | Governed egress               | agentgateway is the single model-credential holder; provider-neutral token alerts + bridge rate limits (§9); agent pods carry only the gateway placeholder key                                                                                                                                                                                                                                                                                                                                                                                                                         |
| agentgateway → self-hosted vLLM                   | Cluster-internal              | Profile-scoped proxy egress; default-deny `models` namespace; serving Pod has no egress; one-shot, prompt-free model loader alone may reach public HTTPS                                                                                                                                                                                                                                                                                                                                                                                                                               |
| Secrets                                           | —                             | Production secrets use SOPS+age; per-service DB roles; registration tokens never appear in git plaintext. The disposable federation lifecycle generates its credentials in-cluster and retains the AgentCard private key only in its bootstrap Secret; public snapshots and workload namespaces receive no private signing material.                                                                                                                                                                                                                                                   |

Standing rules: agent rooms unencrypted **within one org** stays acceptable (ADR 0008) but its rationale **does not survive federation** — revisit is mandatory in M8 (§8.4). New first-party workloads use non-root restricted contexts; several upstream namespaces enforce only PSS `baseline`, so no platform-wide `restricted` claim is made.

The complete assets, actors, boundary diagram, STRIDE pass, control-to-evidence map, and provider/federation/external-bridge residual risks are in the [full threat model](security/threat-model.md). This page remains the stable §7 summary.

## 7.1 Self-hosted model egress proof

The vLLM profile separates supply-chain bootstrap from the prompt data plane. A revision-pinned loader Job may use cluster DNS and public TCP 443 to fetch public model weights into a PVC; it has no Matrix, agent, or prompt input and completes before serving becomes Ready. The serving Pod mounts that cache read-only, disables usage telemetry and online model resolution, and has no NetworkPolicy egress rule. A profile-scoped policy also prevents the agentgateway proxy from reaching anything except cluster DNS, its XDS control plane, vLLM, and the existing kagent A2A backend.

Manifests are intent, not enforcement evidence. K3s normally includes kube-router NetworkPolicy support, but this constrained shared local host aborts `iptables-restore` with `sendmsg() failed: Message too large`; its deny-egress probe remains invalid because no pod firewall chains survive. `mise run test:network-policies:kind` separates that host limitation from manifest correctness: it creates an isolated kind cluster with checksum-pinned Calico, applies the exact policies, proves each required deny/allow path (including vLLM serving egress denial), and verifies that deleting `kagent-allow-platform` turns the test red. This is deterministic enforcement evidence, not acceptance of a separately installed local or GKE CNI. Before claiming that prompts never leave an installed cluster, repeat the active-cluster probe and capture the complete Matrix mention-to-reply flow without external LLM connections.

## 7.2 Attribution is not downstream authentication

The [delegation attribution runbook](audit.md) defines the evidence chain from Matrix sender to the bridge's `X-User-Id` assertion, kagent session/task metadata, agentgateway request logs, and aggregate Prometheus series. A local Synapse event identifies the authenticated Matrix session that submitted it; after the bridge, kagent 0.9.11's unsecure mode accepts the caller-supplied header and marks the resulting user unauthenticated. The bridge/kagent NetworkPolicies are therefore load-bearing controls against a bypassing caller, subject to installed-cluster CNI acceptance in §7.1.

Agentgateway and Prometheus do not currently receive the Matrix event, sender, A2A context/task, or kagent invocation identity on model requests. They corroborate provider/model/token activity but cannot uniquely identify a user or task under concurrency. Treat the bridge's content-free structured record plus kagent's matching session/task as attribution evidence; never treat the model-facing provenance prompt, gateway time correlation, or a high-cardinality metric label as identity proof.

## 7.3 A2A workload authorization

The pinned agentgateway v1.3.1 source defines strict API-key authentication, exposes Secret metadata under `apiKey`, and translates `Require` CEL expressions into fail-closed authorization rules. Fgentic gives only the bridge a random A2A credential; its server-side copy carries `workload: matrix-a2a-bridge`. The policy additionally limits that identity to `POST` under `/api/a2a/kagent/` and AgentCard `GET` requests ending in `/.well-known/agent-card.json`. A missing or invalid credential returns 401; a valid credential with the wrong workload, method, or path returns 403.

This is defense in depth over NetworkPolicy, not a substitute for it. The credential distinguishes the bridge workload from other pods even if ingress isolation is accidentally widened, while NetworkPolicy limits which workloads can attempt a connection and protects kagent's otherwise unauthenticated direct endpoint. One workload-wide bridge credential cannot express a different gateway identity per mapped agent; the bridge mapping and Matrix room membership remain the per-agent authorization boundaries.

`mise run check:a2a-authorization` verifies the manifest, Secret metadata, and CEL contract. `mise run test:a2a-authorization` runs unauthorized, wrong-identity, wrong-path, and valid requests against the exact agentgateway image version pinned by Flux. Target-cluster acceptance still combines that runtime contract with `mise run test:network-policies` so the direct kagent bypass is tested independently.

## 7.4 Governed MCP tool egress

The pinned kagent 0.9.11 source at commit `14dcbfc49bde990a0c737da14a9d94cf7e788303` resolves a `RemoteMCPServer` into the managed Agent pod's runtime configuration. `go/core/internal/controller/translator/agent/adk_api_translator.go` shows that, without `proxy.url`, that pod dials the resource's URL directly; with the configured URL it preserves `/mcp`, changes the destination to the proxy, and adds `x-kagent-host` for the original internal Service. `go/core/internal/controller/reconciler/reconciler.go` separately shows that the controller still dials `kagent-tools:8084` directly to discover tools. The `agent-zoo-egress` policy selects only managed zoo pods, blocks their direct `:8084` path, and retains `:8083` for their required session/persistence API.

agentgateway v1.3.1 at commit `dbaaf7ed73671e7aec9195e35e7f726c0b14b84a` terminates that MCP connection, authenticates platform-helper with a random SOPS-managed bearer key, and binds server-side `apiKey.agent=platform-helper` metadata to it. A fail-closed MCP authorization policy exposes only `k8s_get_resources`, `k8s_describe_resource`, `k8s_get_events`, `k8s_get_resource_yaml`, and `k8s_get_pod_logs`; the upstream server independently runs read-only and has get/list/watch RBAC in six named namespaces with no Secret or cluster-scoped access. docs-qa and scribe have neither a tool declaration nor this credential.

The upstream `kagent-tools` image is immutable by digest. `infra/agentgateway/mcp-surface.pin.json` separately records its container command, ordered runtime arguments, discovery and governed-backend routes, and complete current typed MCP surface: the initialize response plus tools, prompts, resources, and resource templates, with unsupported and supported-empty collections kept distinct. Initialization is pinned because its model-facing `instructions`, server identity, capabilities, metadata, and negotiated protocol can be poisoned before any list call. Every full object is JCS-canonicalized and hashed, so drift in instructions, descriptions, JSON Schemas, annotations, titles, icons, or `_meta` requires an explicit pin update in a reviewed PR. Sorting identities and following every cursor make response ordering irrelevant. The only server-specific normalization sorts the nine identifiers in `k8s_generate_resource.inputSchema.properties.resource_type.description`, which upstream constructs by iterating a Go map; a changed identifier, prose, path, or surrounding schema still produces drift.

The pin deliberately retains `destructiveHint: true` and `openWorldHint: true` on all eight current upstream tools; the typed encoding omits a false/default `readOnlyHint`. Read-only enforcement comes from the immutable `--read-only` process argument, namespaced RBAC, and the five-tool gateway allowlist—not from trusting those annotations.

This key is a narrow, rotatable application credential—not a non-exportable pod identity, mTLS certificate, SPIFFE SVID, Matrix-human identity, or approval token. A compromised platform-helper pod can reuse its own key and any of its five allowed operations. NetworkPolicy does not authenticate a process, and enforcement depends on the target CNI. The controller's direct discovery connection is administrative metadata traffic, not an agent tool call, and remains a separate trusted path. kagent 0.9.11 multiplexes its own `/mcp` handler with session APIs on controller port `8083`; no sample Agent references that handler as a tool, but standard Kubernetes NetworkPolicy cannot allow the required session paths while denying a different HTTP path. Removing that residual reachability needs an upstream separate listener or authenticated session proxy, not an inaccurate L4 policy claim.

For each parsed `tools/call`, agentgateway emits a JSON stdout record marked `audit.kind=fgentic.mcp_tool_call.v1` with authenticated agent, MCP method, resolved tool name/target, status, and duration. The policy adds no arguments, results, prompts, Matrix content, or authorization headers. Kubernetes stdout is not durable retention; production evidence requires an approved restricted sink. Inspect the content-free fields without printing other logs:

```bash
kubectl -n agentgateway-system logs deploy/agentgateway-proxy --since=15m \
  | jq -R '
      fromjson?
      | select(.["audit.kind"] == "fgentic.mcp_tool_call.v1")
      | {
          time,
          agent: .["fgentic.agent"],
          method: .["mcp.method"],
          tool: .["mcp.tool.name"],
          target: .["mcp.tool.target"],
          status: .["http.status"],
          duration
        }'
```

`mise run check:mcp-governance` checks the rendered proxy, credential copies, route, policies, exact tool set, direct-egress omission, audit schema, pin integrity, backend/Service coverage, and complete execution/routing provenance without Docker or a live cluster/server; cold chart or module caches can still require network access. `mise run test:mcp-governance` first compares the live read-only server with that reviewed pin, then uses the exact pinned agentgateway image plus a deliberately write-capable disposable kagent-tools fixture to prove wrong agents are rejected, write tools are filtered, an allowed call reaches the backend, and arguments/credentials stay out of the audit record. `mise run test:network-policies:kind` independently proves the direct managed-agent `:8084` denial and gateway `:8084` allow on Calico. Target-cluster acceptance repeats those paths on the installed CNI, then captures one real platform-helper call and its audit record.

The pin is a review and reconcile boundary, not a content verdict. It cannot detect malicious-from-day-one metadata, prove that implementation behavior matches metadata, or replace authorization and catalog vetting. The official SDK preserves all current typed fields but intentionally drops unknown future typed fields; open maps such as `_meta`, schemas, and experimental or extension capabilities remain complete. Upgrade the versioned pin format when the protocol adds relevant fields. agentgateway v1.3.1 filters and denies MCP identifiers at runtime but has no native committed-hash comparison, and adding an always-on external guardrail only for that comparison would widen the trusted runtime. Scheduled live verification therefore detects post-review drift; #132 remains responsible for first-review evidence.

## 7.5 Cross-organization A2A authorization

The federation profile exposes one deliberate exception to the cluster-internal A2A rule: `POST https://a2a.org-a.fgentic.localhost/api/a2a/kagent/docs-qa`, plus its anonymous AgentCard `GET`. The external Gateway accepts only those exact method/path pairs from `agentgateway-system`; agentgateway handles them on a separate `federation-a2a` listener. The backend remains the internal kagent controller Service, and no route publishes another Agent or exposes kagent directly.

The card and the caller answer different security questions. An ES256 JWS over the RFC 8785 JCS-canonical card, verified with the pinned public P-256 JWK and protected `kid`, establishes the declared docs-qa identity and detects tampering. It does not authorize an invocation. The lab stores that verify-only JWK as a non-secret evidence artifact; production onboarding must deliver and pin it through an independently authenticated channel rather than trusting a key learned from the card endpoint itself. The lab's implemented transport policy instead validates a five-minute org-B `client_credentials` JWT against the exact issuer, remote JWKS, `fgentic-a2a` audience, and `azp=org-b-a2a`. The resulting identity is the OAuth client, not a Matrix MXID, employee, or workload instance. The card stays public so consumers can evaluate identity and capabilities before authenticating.

For every `SendMessage`, authorization also requires the exact `A2A-Version: 1.0` service parameter, the token-budget extension, and an integer `maxTokens` within the configured ceiling. The version guard prevents a v1 card from silently invoking kagent's compatibility-default v0.3 JSON-RPC binding. The global rate-limit service uses the verified `azp` as its descriptor and `maxTokens` as admission cost, failing closed when quota cannot be checked. This reserves a caller-declared maximum; it is not provider-reported consumption. Actual model-token metrics arise on the later kagent → agentgateway model hop and remain aggregate, so they cannot support per-consumer billing or prove which A2A task used a token.

mTLS is the supported alternative when partners operate certificate identity rather than OIDC: terminate on a dedicated client-certificate listener, trust only the partner CA, map an exact SAN to the quota consumer, and define issuance, rotation, and revocation. The lab does not implement or test that path, and ordinary server-side TLS is not mTLS. Whichever pattern a deployment selects, keep the exact route, fail-closed policy, per-consumer quota, and direct-kagent NetworkPolicy invariant.

## 7.6 Admission-enforced platform invariants

The cluster installs `infra/policies/` before creating any managed namespace. Kubernetes' in-process ValidatingAdmissionPolicy then rejects four classes of drift without adding a webhook controller:

1. Pods in a namespace labelled `fgentic.dev/managed=true` cannot use an explicit `:latest` tag or an untagged reference that implicitly resolves to latest in a normal, init, or ephemeral container.
1. A managed namespace must declare `enforce`, `audit`, and `warn` Pod Security levels plus its image-policy phase. PSS version labels are forbidden so the levels always use the current standard. The PSS values and managed marker cannot later be changed or removed through the API; image policy may move from `audit` to `enforce`, but never backwards.
1. A `kagent.dev/v1alpha2` Agent must live in `kagent`, be Declarative, use only `agentgateway-model`, keep the Python runtime, and run under the reviewed `agent-zoo-runtime` pod identity without image, container, environment, volume, scheduling, security-context, or service-account overrides. Secondary memory/compaction models and Secret-backed prompts are rejected. Only `platform-helper` may declare a tool, and that one reference is limited to the reviewed same-namespace RemoteMCPServer, five read-only tool names, and its exact Authorization Secret. Inline credentials and request-header propagation are rejected. BYO, Agent-as-tool, sandbox, legacy top-level skills, and model-generated code execution are outside the admitted surface.
1. Managed namespaces admit only ClusterIP or ExternalName Services. The exact `gateway/traefik` LoadBalancer is the sole public-service exception; NodePort and `externalIPs` are rejected everywhere else.

The digest rule is deliberately staged rather than falsely platform-wide. The `models` and optional `bridges` namespaces contain only digest-pinned owned workloads and use `fgentic.dev/image-policy=enforce`, so tag-only Pods are denied. Other namespaces use `audit`: upstream Helm charts still emit tag-only image references, and the supported local bridge side-load is intentionally `matrix-a2a-bridge:local`. Those requests receive a client warning and an audit annotation; they must move to `enforce` only after every controller-generated Pod survives a fresh rollout. Audit annotations become durable evidence when Kubernetes API audit logging lands (#161); `Warn` keeps the debt visible before then.

Every CEL policy uses `failurePolicy: Fail`, and its annotation records the simple constant or linear cost shape. Namespace selectors exclude `default`, `flux-system`, `kube-system`, `kube-public`, and `kube-node-lease` because Fgentic does not label or claim ownership of them. This is scoping, not a bypass for platform workloads. API-created admission policies cannot intercept mutation of ValidatingAdmissionPolicy or its bindings themselves; Flux drift detection and repository review remain that control. CEL also cannot prove cross-object invariants such as a namespace having a NetworkPolicy, so the independent conformance gate remains load-bearing.

`mise run check:admission-policies` checks the policy set, staged bindings, managed namespace profiles, and policy-before-namespace Flux edge offline. With an explicitly approved Kubernetes 1.30+ test context, `ADMISSION_POLICY_CONTEXT=<context> mise run test:admission-policies` performs server-side negative probes without starting or pulling a workload: missing or weakened PSS labels, explicit or implicit latest images across every container class, tag-only images in enforce mode, NodePort, `externalIPs`, wrong Agent models, pod overrides, Agent-as-tool, MCP target/tool expansion, request-header propagation, and credential substitution are rejected; tag-only audit mode warns; pinned Pods, ClusterIP, and all three repository Agents pass. `demo:up` runs that same contract after a clean reconcile and before the mention-to-reply seed.

MutatingAdmissionPolicy is intentionally absent while the k3s pin remains 1.35; reconsider its stable API after moving to Kubernetes 1.36. Admission mutation would complement these validations, not replace explicit GitOps manifests.
